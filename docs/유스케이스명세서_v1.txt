UC-U01: QR로 워크숍 접속하기
────────────────────────────────────────────────────────────────

  ID:           UC-U01
  이름:         QR로 워크숍 접속하기
  액터:         사용자
  우선순위:     P1 필수
  대응 페르소나: 김하늘(축제), 박소연(기획)
  사용 환경:    A(축제 현장)

  ■ 사전 조건
  - 사용자가 스마트폰을 소지하고 있다
  - QR코드가 인쇄된 포스터가 설치되어 있다
  - 모바일 네트워크(LTE/5G/Wi-Fi)가 연결되어 있다

  ■ 기본 흐름
  1. 사용자가 축제 부스에서 QR 포스터를 발견한다
  2. 사용자가 스마트폰 카메라로 QR코드를 스캔한다
  3. 시스템이 모바일 브라우저에서 행사 전용 랜딩(/event/행사명.html)을 로딩한다
  4. 시스템이 1초 이내에 페이지를 완전히 표시한다
  5. 캐릭터 마음이가 bounce 애니메이션과 함께 등장한다
  6. 마음이가 "안녕! 축제에 온 걸 환영해. 5분만 나를 만나보는 건 어때?"라고 안내한다
  7. 사용자가 추천 워크숍 목록에서 하나를 선택한다
  8. → UC-U04 또는 UC-U05로 진행

  ■ 대안 흐름
  A1. QR 인식 실패 시
      1. 사용자가 QR 아래 짧은 URL(bit.ly/마음워크숍)을 직접 입력한다
      2. 기본 흐름 3번으로 합류

  A2. 행사 전용 랜딩이 아닌 메인 페이지로 연결된 경우
      1. → UC-U03으로 진행

  ■ 사후 조건
  - 사용자가 워크숍 체험을 시작할 수 있는 상태
  - 앱 설치, 회원가입, 개인정보 입력 없이 도달

  ■ 비기능 요구사항
  - 로딩 시간: 1초 이내 (HTML-only, 초경량)
  - 네트워크: 불안정 환경(축제 야외)에서도 작동
  - 앱 설치: 요구하지 않음
  - HTTPS: 필수 (학교 Wi-Fi 보안 정책 대응)

UC-U02: URL 링크로 워크숍 접속하기
────────────────────────────────────────────────────────────────

  ID:           UC-U02
  이름:         URL 링크로 워크숍 접속하기
  액터:         사용자
  우선순위:     P1 필수
  대응 페르소나: 이준호(카톡), 최민수(에브리타임), 윤서현(인스타)
  사용 환경:    B(개인 여유), C(직장 점심), E(밤 혼자)

  ■ 사전 조건
  - 사용자가 SNS/메신저/게시판에서 워크숍 URL을 발견했다
  - 또는 이전 방문 북마크가 있다

  ■ 기본 흐름
  1. 사용자가 URL 링크를 탭/클릭한다
  2. 시스템이 해당 워크숍 페이지를 모바일 브라우저에 로딩한다
  3. 시스템이 1초 이내에 페이지를 표시한다
  4. → UC-U04 / UC-U05 / UC-U06으로 진행 (워크숍 종류에 따라)

  ■ 대안 흐름
  A1. OG 미리보기 카드를 통한 접속
      1. SNS에서 OG 미리보기(제목+설명+캐릭터 이미지)를 확인한다
      2. 미리보기 카드를 탭한다
      3. 기본 흐름 2번으로 합류

  A2. 북마크를 통한 메인 접속
      1. → UC-U03으로 진행

  ■ 사후 조건
  - 사용자가 해당 워크숍 도입 화면을 보고 있는 상태

UC-U03: 메인에서 워크숍 선택하기
────────────────────────────────────────────────────────────────

  ID:           UC-U03
  이름:         메인에서 워크숍 선택하기
  액터:         사용자
  우선순위:     P1 필수
  대응 페르소나: 전체

  ■ 사전 조건
  - 사용자가 index.html(메인 페이지)에 접속해 있다

  ■ 기본 흐름
  1. 시스템이 히어로 영역에 캐릭터 마음이(64px)와 슬로건을 표시한다
  2. 시스템이 미니 워크숍 섹션(2열 카드)을 표시한다
     - 각 카드에 [선택형] [혼합형] 등 체험 유형 뱃지 표시
  3. 시스템이 풀 워크숍 섹션(리스트 형태)을 표시한다
  4. 사용자가 원하는 워크숍을 탭한다
  5. 시스템이 해당 워크숍 페이지로 이동한다
  6. → UC-U04 / UC-U05 / UC-U06으로 진행

  A1. 카테고리 필터링 → UC-U15

  ■ 사후 조건
  - 사용자가 선택한 워크숍의 도입 화면에 도달

UC-U04: 미니 워크숍 체험하기 (유형 A: 선택형)
────────────────────────────────────────────────────────────────

  ID:           UC-U04
  이름:         미니 워크숍 체험하기 (유형 A)
  액터:         사용자
  우선순위:     P1 필수
  대응 페르소나: 김하늘(축제), 이준호(동료 공유), 박소연(기획 검토)
  사용 환경:    A(축제), B(개인), C(직장)
  소요 시간:    약 1분

  ■ 사전 조건
  - UC-U01/U02/U03을 통해 미니 워크숍 페이지에 도달했다

  ■ 기본 흐름

  [시작]
  1. 시스템이 도입 화면을 표시한다
     - 캐릭터 마음이(반가움 표정)가 bounce 애니메이션으로 등장
     - 말풍선: 도입 대사 + Permission 메시지
       예: "안녕! 1분이면 돼. 어떤 결과든 괜찮아."
     - 안전 공간 안내: "답은 저장되지 않아요." (12px, #BBB)
  2. 사용자가 "같이 해보자! →" 버튼을 탭한다
  3. 시스템이 #q1 앵커로 이동한다 (:target 전환)

  [질문 1~N]
  4. 시스템이 질문과 선택지(radio 또는 리커트 또는 카드)를 표시한다
     - 캐릭터 마음이(경청 표정) + 안내 대사
       예: "가장 가까운 걸 골라봐."
  5. 사용자가 선택지 중 하나를 탭한다
  6. 시스템이 해당 radio를 :checked 상태로 변경한다
  7. 시스템이 선택에 따른 캐릭터 반응을 표시한다
     - 긍정 선택 → 마음이(응원): "좋은 아침이었나 보다!"
     - 중립 선택 → 마음이(기본): "그렇구나."
     - 부정 선택 → 마음이(공감): "좀 피곤한 아침이었나 봐."
     - fadeIn 0.3s 애니메이션
  8. 사용자가 "다음 →" 버튼을 탭한다
  9. 시스템이 다음 질문(#q2)으로 :target 전환한다
  10. 4~9를 나머지 질문에 대해 반복한다

  [결과]
  11. 마지막 질문 후 사용자가 "결과 보기 →"를 탭한다
  12. 시스템이 #result로 :target 전환한다
  13. → UC-U07로 진행

  ■ 대안 흐름
  A1. 리커트 척도 질문인 경우 (M03, M04, M07)
      4a. 시스템이 5점 척도(전혀아니다~매우그렇다)를 가로 배치한다
      5a. 사용자가 5개 원형 버튼 중 하나를 탭한다
      6a~7a. 기본 흐름과 동일

  A2. 카드 2×2 질문인 경우 (M05, M08)
      4b. 시스템이 2×2 카드 그리드(이모지+텍스트)를 표시한다
      5b. 사용자가 카드 하나를 탭한다
      6b. 시스템이 해당 카드에 보더+그림자 강조 표시
      7b. 캐릭터 반응 표시

  A3. 사용자가 중간에 [홈] 탭 시
      1. 시스템이 index.html로 이동한다
      2. 체험 내용은 사라진다 (데이터 미저장)

  ■ 사후 조건
  - 시스템이 :checked 상태의 조합에 따라 결과를 표시할 준비 완료

UC-U05: 풀 워크숍 체험하기 (유형 B: 혼합형)
────────────────────────────────────────────────────────────────

  ID:           UC-U05
  이름:         풀 워크숍 체험하기 (유형 B)
  액터:         사용자
  우선순위:     P1 필수
  대응 페르소나: 이준호(점심시간), 최민수(밤), 김하늘(재방문)
  사용 환경:    B(개인), C(직장), E(밤)
  소요 시간:    약 3~5분

  ■ 사전 조건
  - UC-U01/U02/U03/U11을 통해 풀 워크숍 페이지에 도달했다

  ■ 기본 흐름

  [STEP 1 도입]
  1. 시스템이 도입 화면을 표시한다
     - 주제 이모지(48px) + pulse 애니메이션
     - 캐릭터 마음이(반가움) + 도입 대사
       예: "하루에도 감정은 여러 번 변해.
            어떤 날씨든 괜찮아."
     - "3분이면 돼 · 답은 저장되지 않아요"
     - 진행률 바: ━━━━━━ 1/4
  2. 사용자가 "같이 해보자! →"를 탭한다

  [STEP 2 탐색]
  3. 시스템이 탐색 화면을 표시한다
     - 캐릭터 마음이(경청): "천천히 살펴봐. 정답은 없어."
     - <details><summary> 카드 3~4장
  4. 사용자가 <details> 카드를 탭하여 내용을 살펴본다
  5. 사용자가 "다음 →"을 탭한다

  [STEP 3 체험]
  6. 시스템이 체험 화면을 표시한다
     - 캐릭터 마음이: "감정 날씨를 골라봐."
  7. 사용자가 선택지(radio)를 탭한다
  8. 시스템이 :checked 반응 + 캐릭터 반응을 표시한다
  9. (슬라이더가 있는 경우) 사용자가 range 슬라이더를 조작한다
  10. 시스템이 서술 영역을 표시한다
      - 캐릭터 마음이(경청): "적고 싶은 게 있으면 적어봐도 좋아.
                              안 적어도 괜찮아. 여기 적은 건 아무도 안 봐."
      - <textarea> placeholder: "(한 줄이라도 괜찮아)"
  11. 사용자가 서술을 적거나, 적지 않고 넘어간다
  12. 사용자가 "결과 보기 →"를 탭한다

  [STEP 4 결과]
  13. → UC-U07로 진행

  ■ 대안 흐름
  A1. 사용자가 서술을 적지 않는 경우
      - "결과 보기" 버튼은 항상 활성 상태
      - 서술 여부와 관계없이 결과 정상 표시

  ■ 사후 조건
  - 사용자가 결과 카드를 확인할 수 있는 상태
  - 서술 내용은 어디에도 저장되지 않음

  [STEP 3 체험 — 탭 전환]

     - 캐릭터 마음이: "비판자에게 한마디 해보자!"
  7. 시스템이 "직접 쓰기 / 골라 쓰기" 탭을 표시한다
     - 기본 선택: "골라 쓰기" 활성 (서술 부담 방지)
  8-a. [골라 쓰기 경로]
     8a-1. 사용자가 "☑ 골라 쓰기" 탭을 확인한다 (기본 활성)
     8a-2. 시스템이 4~5개 롤모델 메시지 선택지(radio)를 표시한다
           예: ○ "고마워, 하지만 이제는 내가 결정할게."
               ○ "지금은 좀 쉬어도 괜찮아. 나는 충분히 하고 있어."
     8a-3. 사용자가 하나를 선택한다
     8a-4. 시스템이 캐릭터 반응을 표시한다
           마음이(공감): "좋은 말이다. 그래, 충분히 하고 있어."
     8a-5. 시스템이 "추가로 적고 싶은 게 있으면" 안내와 작은 textarea를 표시한다
     8a-6. 사용자가 추가 서술을 적거나 적지 않는다
  8-b. [직접 쓰기 경로]
     8b-1. 사용자가 "✍ 직접 쓰기" 탭을 탭한다
     8b-2. 시스템이 CSS :checked로 탭 전환하여 textarea를 표시한다
     8b-3. 캐릭터 마음이(경청): "자유롭게 적어봐. 한 글자도 괜찮아.
                                 여기 적은 건 아무도 안 봐."
     8b-4. 사용자가 자유 서술을 작성한다
  9. 사용자가 "결과 보기 →"를 탭한다

  10. → UC-U07로 진행

UC-U07: 결과 카드 확인하기
────────────────────────────────────────────────────────────────

  ID:           UC-U07
  이름:         결과 카드 확인하기
  액터:         사용자
  우선순위:     P1 필수
  대응 페르소나: 전체

  ■ 사전 조건
  - UC-U04/U05/U06에서 "결과 보기"를 탭한 상태

  ■ 기본 흐름
  1. 시스템이 :checked 상태의 조합에 따라 해당 결과 영역을 표시한다
     (CSS :checked + 형제 선택자로 결과 분기)
  2. 시스템이 결과 카드를 slideUp 애니메이션으로 표시한다
     - 정사각형(1:1) 카드
     - 카테고리 연한 배경색
     - 결과 수치/문구 + 이모지
     - 캐릭터 마음이(36px, 해당 표정) 카드 내 포함
     - CSS 별/반짝이 장식
     - 하단 URL 워터마크
  3. 시스템이 결과 카드 아래에 캐릭터 검열자 대응 대사를 표시한다
     - 긍정 결과: 마음이(응원) "대단해! 오늘 기분 최고네 ☀"
     - 부정 결과: 마음이(위로) "여기까지 온 것만으로도 나를 돌보고 있는 거야."
  4. 시스템이 공유 영역을 표시한다
     - 스크린샷 안내 + [이미지 저장] [링크 공유] 버튼
     - "캡처해서 저장하거나 공유해봐! 안 해도 괜찮아."
  5. 시스템이 실천 가이드를 표시한다
     - 캐릭터 마음이: "이건 그냥 제안이야. 하고 싶은 것만 해봐."
     - 구체적 행동 3가지 (아티스트 데이트 요소 포함)
  6. 시스템이 다른 워크숍 추천 2개를 표시한다
     - 각 추천에 체험 유형 뱃지 표시
  7. 시스템이 위기 상담 안내 푸터를 표시한다 (캐릭터 미등장)

  ■ 대안 흐름
  A1. 매우 부정적 결과인 경우
      3a. 캐릭터 위로 대사가 더 길고 따뜻하게 표시된다
      3b. 실천 가이드가 더 작은 행동으로 구성된다
          예: "창문 열고 5분 바깥 바라보기"
      3c. 전문 상담 안내가 자연스러운 톤으로 실천 가이드 뒤에 추가 배치된다
          "더 깊은 이야기를 나누고 싶다면, 전문 상담사와 대화해보는 것도 좋아."

  A2. 미선택으로 결과 보기한 경우
      1a. showResult()가 표시 중인 결과 카드 없음을 감지한다
      1b. 시스템이 마지막 질문의 "잘 모르겠어" radio를 자동 체크한다
      1c. "모르겠어" 결과 카드가 표시된다 (🤷, 공감 메시지)
      1d. 이미지 저장·공유 모두 정상 작동한다
      → 사용자는 빈 화면을 절대 보지 않음
      → "모르겠는 것도 자연스러운 거야" 메시지로 안전하게 종료

  ■ 사후 조건
  - 사용자가 결과를 확인하고, 공유/저장/전환 행동을 선택할 수 있는 상태
  - 미선택 시에도 반드시 결과 카드가 표시된 상태 (빈 화면 없음)

  대응 페르소나: 최민수(위기 상황)

  ■ 사전 조건
  - 사용자가 어떤 페이지에든 있다 (모든 페이지에 푸터 상시 표시)

  ■ 기본 흐름
  1. 사용자가 페이지 하단 위기 상담 안내 영역을 확인한다
     "지금 힘든 순간이라면, 혼자 감당하지 않아도 됩니다."
  2. 사용자가 전화번호(1393 / 1577-0199 / 1388)를 탭한다
  3. 시스템이 <a href="tel:1393">을 통해 전화 앱을 연다
  4. 사용자가 전화를 건다

  ■ 비기능 요구사항
  - 위기 안내 영역에 캐릭터를 배치하지 않는다 (진지함 유지)
  - 모든 페이지 하단에 반드시 노출 (예외 없음)
  - 전화번호는 <a href="tel:"> 링크로 구현 (모바일 원탭 전화)

────────────────────────────────────────────────────────────────
UC-O01: 워크숍 콘텐츠 추가하기
────────────────────────────────────────────────────────────────

  ID:           UC-O01
  이름:         워크숍 콘텐츠 추가하기
  액터:         운영자 (1인 개발자)
  우선순위:     P1 필수
  소요 시간:    2~3시간

  ■ 기본 흐름
  1. 운영자가 템플릿 HTML 파일을 복사한다
     - 유형 A: /workshop/template-A.html
     - 유형 B: /workshop/template-B.html
     - 유형 C: /workshop/template-C.html
     - 미니: /mini/template.html
  2. 운영자가 내용을 교체한다
     - 워크숍 제목, 카테고리, 이모지
     - 캐릭터 대사 세트 (도입/반응/결과)
     - 질문·선택지 텍스트 및 radio/checkbox 구조
     - (유형 C인 경우) 골라 쓰기 선택지 4~5개
     - 결과 분기 메시지 (:checked 조합별)
     - 실천 가이드 3가지
     - 다음 추천 워크숍 링크
     - OG 메타 태그 (title, description, image)
  3. 운영자가 결과 카드 색상(CSS class)을 설정한다
  4. 운영자가 OG 이미지를 Canva로 제작한다 (캐릭터 포함)
  5. 운영자가 검수 체크리스트를 확인한다
     □ 기능 검수 (선택 조합별 결과, :target/:checked 작동)
     □ 선택지 검수 (리커트 터치, 카드 레이아웃, 탭 전환)
     □ 캐릭터 검수 (대사 톤, 표정 전환, 위기 안내 미등장)
     □ AW 검수 (Permission, 안전 공간, 검열자 대응, 실천 가이드)
     □ 모바일 테스트 (아이폰 + 안드로이드)
  6. 운영자가 index.html 목록에 새 워크숍 링크를 추가한다
  7. → UC-O04(배포)로 진행

  ■ 사후 조건
  - 새 워크숍 HTML 파일이 완성되어 배포 대기 상태

UC-O03: QR코드 생성하기
────────────────────────────────────────────────────────────────

  ID:           UC-O03
  이름:         QR코드 생성하기
  액터:         운영자
  우선순위:     P2
  소요 시간:    10분

  ■ 기본 흐름
  1. 운영자가 행사 전용 랜딩 URL을 확인한다
  2. 운영자가 무료 QR 생성기에서 QR 이미지를 생성한다
  3. 운영자가 짧은 URL(bit.ly 등)도 생성한다 (QR 인식 실패 대비)
  4. 운영자가 QR + 짧은 URL + 캐릭터가 포함된 포스터를 Canva로 디자인한다
  5. 운영자가 포스터 PDF를 기획자에게 전달하거나 직접 인쇄한다

  ■ 사후 조건
  - QR 포스터가 인쇄 가능한 상태

UC-O04: 워크숍 배포하기
────────────────────────────────────────────────────────────────

  ID:           UC-O04
  이름:         워크숍 배포하기
  액터:         운영자
  우선순위:     P1 필수
  소요 시간:    5분

  ■ 기본 흐름
  1. 운영자가 변경된 파일을 git add로 스테이징한다
  2. 운영자가 git commit으로 커밋한다
  3. 운영자가 git push로 원격 저장소에 푸시한다
  4. GitHub Pages가 자동으로 배포한다
  5. 운영자가 배포된 URL에 접속하여 정상 작동을 확인한다
  6. 운영자가 OG 미리보기를 카카오톡 링크로 테스트한다

  ID:           UC-O05
  이름:         피드백 수집하기
  액터:         운영자
  우선순위:     P2

  ■ 기본 흐름
  1. 운영자가 구글 폼 피드백 응답을 확인한다
  2. 운영자가 SNS 해시태그(#마음워크숍)를 검색하여 반응을 수집한다
