14. JSON 분리 vs HTML 하드코딩 비교 분석
────────────────────────────────────────────────────────────────

  ■ 9개 관점 비교

  관점                HTML 하드코딩           JSON 분리
  ═══════════════════════════════════════════════════════════════
  1. 42개 확장       파일 42개 복사           JSON 42개 + 렌더러 1개
                     구조 다양성 수용 용이    스키마 통일 어려움

  2. 1인 유지보수    HTML만 수정             JSON + 렌더러 양쪽 수정
                     브라우저에서 즉시 확인   JS 디버깅 필요

  3. Minimal-JS      result.js 7KB만          +렌더러 800줄 이상
                     원칙 유지               원칙 심각히 위반

  4. CSS :checked    HTML에서 형제 관계 보장  JS가 동적 생성 →
                     눈으로 확인 가능        순서 실수 시 깨짐

  5. 성능            즉시 파싱·렌더링        fetch → 파싱 → DOM 생성
                     FCP 최소                FCP 지연

  6. 오프라인        브라우저 캐시로 작동     fetch 실패 대응 필요
                     추가 코드 불필요        에러 핸들링 필수

  7. 제작 효율       복사 2~3시간/개         30분~1시간/개
                     템플릿 정비로 개선 가능  렌더러 개발 2~3일 선투자

  8. SEO/OG          <head>에 하드코딩 → 즉시 JS 렌더링 → 크롤러 못 읽음
                     카카오/트위터 정상       공유 미리보기 깨짐
                                             SSR/SSG 필요 → 복잡도↑

  9. 빌드 도구       불필요 (원본=배포본)     사실상 필수
                     git push = 배포         빌드 → 결과물 → 배포
  ═══════════════════════════════════════════════════════════════

  ■ 정량 비교

  항목              HTML 하드코딩           JSON 분리
  ═══════════════════════════════════════════════════════════════
  JS 코드량          336줄 (현재)            800~1200줄 예상
  초기 구축 비용     0시간 (현재 완료)       2~3일 (렌더러 개발)
  워크숍 1개 제작    2~3시간 (개선 후 1시간)  30분~1시간
  37개 추가 제작     37~74시간               18~37시간
  렌더러 유지보수    해당 없음               지속적 (유형 추가마다)
  SEO 대응           추가 작업 없음          SSG 도입 필요 (+20시간)
  손익분기점         -                       불확실 (유형 다양성)

────────────────────────────────────────────────────────────────
15. 최종 결정: HTML 하드코딩 유지
────────────────────────────────────────────────────────────────

  ★ HTML 하드코딩을 유지하되, 템플릿 체계를 정비하여 복사 효율 개선

  근거:
  [1] Minimal-JS 원칙 유지 — JSON 렌더러는 원칙에 심각히 위반
  [2] CSS :checked 안정성 — 형제 관계를 HTML에서 직접 보장
  [3] SEO/OG 즉시 작동 — 크롤러가 <head> 메타를 바로 읽음
  [4] 오프라인 안정성 — fetch 실패 걱정 없음
  [5] 빌드 도구 불필요 — "원본 = 배포본" 유지
  [6] 1인 개발자 인지 부하 — HTML만 알면 됨
  [7] 워크숍 유형 다양성 — 통일 스키마보다 개별 HTML이 유연

  대신 개선할 것:
  → 유형별 템플릿 3종을 정비하여 복사 효율 2~3시간 → 1시간으로 단축

  ■ 템플릿 3종

  파일명                     유형    대표 워크숍        교체 항목 수
  ═══════════════════════════════════════════════════════════════

  ★ 강점발견(checkbox 유형)은 구조가 특수하여
    template-radio를 복사 후 수동 수정

  ■ 템플릿 내 교체 마커 [v2 신설]

  템플릿에 교체할 위치를 주석 마커로 표시:

  <!-- @@TITLE@@ --> → 워크숍 제목
  <!-- @@EMOJI@@ --> → 주제 이모지
  <!-- @@CATEGORY@@ --> → body class (category-emotion 등)
  <!-- @@DURATION@@ --> → 소요시간 (1분/3분)
  <!-- @@INTRO_SPEECH@@ --> → 캐릭터 도입 대사
  <!-- @@PERMISSION@@ --> → Permission 메시지
  <!-- @@Q1_TEXT@@ --> → Q1 질문 텍스트
  <!-- @@Q1_OPTIONS@@ --> → Q1 선택지 HTML 블록
  <!-- @@RESULT_HIGH@@ --> → 긍정 결과 카드 HTML
  <!-- @@RESULT_MID@@ --> → 중립 결과 카드 HTML
  <!-- @@RESULT_LOW@@ --> → 부정 결과 카드 HTML
  <!-- @@RESULT_IDK@@ --> → 모르겠어 결과 카드 HTML
  <!-- @@GUIDE1@@ --> → 실천 가이드 1
  <!-- @@GUIDE2@@ --> → 실천 가이드 2
  <!-- @@GUIDE3@@ --> → 실천 가이드 3
  <!-- @@NEXT1_URL@@ --> → 추천 워크숍 1 링크
  <!-- @@NEXT1_TITLE@@ --> → 추천 워크숍 1 제목
  <!-- @@NEXT2_URL@@ --> → 추천 워크숍 2 링크
  <!-- @@NEXT2_TITLE@@ --> → 추천 워크숍 2 제목
  <!-- @@OG_TITLE@@ --> → OG 제목
  <!-- @@OG_DESC@@ --> → OG 설명
  <!-- @@SHARE_URL@@ --> → 공유 URL


  ■ 워크숍 추가 워크플로우 (v2 개선)

  1. 유형에 맞는 템플릿 복사 (template.html → 새이름.html)
  2. 마커(@@...@@)를 검색하여 내용 교체
  3. 결과 카드 색상 CSS class 확인 (card-emotion 등)
  4. OG 메타 태그 교체
  5. 모바일 테스트 (브라우저에서 바로 열기)
  6. git push → 배포

  예상 소요: 30분~1시간 (기존 2~3시간에서 단축)

21. ADR-012 콘텐츠 저장 방식 결정
────────────────────────────────────────────────────────────────

  ADR-012: HTML 하드코딩 유지 (JSON 분리 미채택)
  ────────────────────────────────────────────
  결정: 콘텐츠를 JSON 파일로 분리하지 않고
       HTML 파일에 하드코딩하는 방식을 유지한다.

  근거:
  · JSON 렌더러 개발 시 JS 800줄+ → Minimal-JS 원칙 위반
  · CSS :checked 형제 관계가 JS 동적 생성에서 불안정
  · SEO/OG 메타가 크롤러에 바로 읽혀야 함 (JS 렌더링 불가)
  · 오프라인 fetch 실패 대응 불필요
  · 빌드 도구 없이 "원본 = 배포본" 유지
  · 워크숍 유형 다양성으로 통일 스키마 비현실적

  트레이드오프:
  · 워크숍 추가 시 HTML 복사+교체 필요 (1시간/개)
  · 전체 워크숍 일괄 수정 어려움 (예: 위기 안내 문구 변경)
  → 템플릿 체계 정비 + 검색/바꾸기로 완화

  대안으로 검토했으나 미채택한 것:
  · JSON + fetch → JS 의존도 과다
  · JSON + SSG(빌드) → 복잡도 과다 (1인 개발)
  · JSON + 인라인 삽입 → SEO는 해결되나 빌드 필요
